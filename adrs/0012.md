# ADR012: Implementación del Patrón DTO para Transferencia de Datos
**Estado:** Aceptada  
**Participantes:** Federica Cabrera y Ana Gutman  

## Contexto
La aplicación necesita una forma eficiente de transferir datos entre el cliente y el servidor, con el objetivo de reducir la complejidad y aumentar el rendimiento. Dado que las respuestas de las API pueden ser grandes y complejas, es necesario un mecanismo que permita transferir solo los datos relevantes de manera estructurada, sin exponer la lógica interna o la complejidad del modelo de dominio.

## Decisión
Se ha decidido implementar el **patrón DTO (Data Transfer Object)** para la transferencia de datos entre el cliente y el servidor. Este patrón permite crear objetos simples que contienen solo los datos necesarios para ser enviados o recuperados del servidor.

### Descripción de la Implementación:
- **Creación de objetos simples**: El DTO se diseñará como un objeto plano que encapsula una serie de atributos, los cuales pueden ser enviados o recuperados del servidor en una única invocación.
- **Consolidación de información de múltiples fuentes**: Un DTO puede contener datos de diferentes fuentes o tablas, consolidando la información necesaria en una única clase simple para evitar múltiples solicitudes o procesos complejos en el servidor.
- **Desacoplamiento entre el modelo de dominio y la capa de presentación**: El patrón DTO permite desacoplar el modelo de dominio de la forma en que los datos son presentados al usuario. Esto mejora la flexibilidad al modificar la estructura interna de la aplicación sin afectar las respuestas enviadas al cliente.
  
## Consecuencias
- **Beneficios:**  
  - **Reducción de la complejidad en las respuestas**: Al consolidar múltiples datos en un único objeto DTO, se reduce la cantidad de objetos y solicitudes requeridas entre el servidor y el cliente, mejorando el rendimiento y reduciendo la latencia.
  - **Desacoplamiento**: El uso del patrón DTO ayuda a mantener un diseño más limpio, permitiendo que el modelo de dominio se modifique sin afectar directamente a la API o la capa de presentación.  
  - **Flexibilidad**: Permite la inclusión solo de los datos necesarios en las respuestas API, lo que facilita la personalización según las necesidades del cliente sin modificar el modelo de dominio.
  - **Mejora en la seguridad**: Al ocultar la complejidad interna y no exponer directamente los objetos de dominio, se reduce el riesgo de exponer información sensible o innecesaria.
  
  El patrón DTO proporciona una solución estructurada y eficiente para la transferencia de datos entre el cliente y el servidor, lo cual es clave para optimizar el rendimiento de la aplicación y mantener la flexibilidad del sistema. Al desacoplar la capa de presentación de la lógica interna y consolidar datos en objetos simples, el uso del patrón DTO no solo mejora el rendimiento y la seguridad, sino que también facilita la escalabilidad del sistema a medida que crece la complejidad de los datos manejados.

- **Costos:**  
  - **Sobrecarga en el diseño**: La implementación del patrón DTO requiere el diseño de clases adicionales para cada conjunto de datos que se transfiera. Esto puede aumentar la complejidad del código en comparación con el uso directo de entidades del modelo.
  - **Mantenimiento adicional**: A medida que evoluciona el modelo de dominio, los DTO también deben actualizarse para reflejar los cambios, lo que puede generar trabajo adicional en la gestión del código.
