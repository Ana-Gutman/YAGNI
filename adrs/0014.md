# ADR013: Rechazo de Interfaces para los ABM (Alta, Baja, Modificación)
**Estado:** Rechazada  
**Participantes:** Ana Gutman, Federica Cabrera  

## Contexto
En el proyecto se evaluó la opción de utilizar **interfaces** para los ABM (Alta, Baja, Modificación) de las entidades del sistema, con el objetivo de promover la flexibilidad, la reutilización del código y la extensibilidad del sistema. Sin embargo, tras una evaluación más profunda de las necesidades actuales y los objetivos de la arquitectura, se decidió que el uso de interfaces no aportaría un valor significativo en este caso.

## Decisión
Se ha decidido no implementar **interfaces** para las operaciones de ABM en las entidades del sistema. La razón principal fue que no existe una necesidad inmediata de abstracción que justifique el uso de interfaces para estas operaciones, lo que podría introducir complejidad innecesaria.

### Justificación:
- **Simplicidad en la implementación**: El uso de interfaces para los ABM puede ser útil en sistemas que requieren una gran flexibilidad o extensibilidad, como en los casos donde las operaciones de ABM se deben personalizar o adaptar a varios tipos de entidades. Sin embargo, en este proyecto, las operaciones de ABM son simples y no requieren una flexibilidad adicional.
  
- **Reducción de la complejidad**: Implementar interfaces para cada una de las operaciones ABM añade un nivel adicional de abstracción que no es necesario en el contexto actual. Las implementaciones actuales son suficientemente claras y directas, sin la necesidad de un contrato de interfaz que agregue más código y reduce la transparencia.

- **Mantenimiento de la coherencia**: Crear interfaces para los ABM puede dificultar la comprensión del sistema al introducir más puntos de contacto entre diferentes partes del código, lo que no sería beneficioso cuando la estructura del sistema es sencilla y no requiere extensiones dinámicas.

- **Enfoque práctico**: Las entidades del sistema tienen operaciones de ABM simples, y la necesidad de modificar la lógica detrás de estas operaciones es poco probable a corto plazo. Implementar interfaces resultaría en más trabajo sin un beneficio claro.

## Consecuencias
- **Beneficios:**  
  - **Menor complejidad**: Al no utilizar interfaces para los ABM, el código se mantiene más sencillo y fácil de entender, lo cual facilita su mantenimiento y evolución.
  - **Mayor claridad en la implementación**: Las funciones de ABM son directas y no requieren una capa adicional de abstracción, lo que facilita la comprensión del flujo de datos y la lógica de negocio.

- **Costos:**  
  - **Menor flexibilidad en el futuro**: Si en el futuro se requiere una mayor personalización en los ABM, se tendría que refactorizar el sistema para introducir interfaces y abstracciones adicionales.
  - **Posible repetición de código**: Al no abstraer las operaciones de ABM a través de interfaces, podría haber una ligera duplicación de código en diferentes partes del sistema si se repiten las mismas operaciones, aunque en el contexto actual esto no es un problema significativo.
